## Пропозиція реалізації на Next.js (багатошкільна версія)

### 1. Загальна концепція

**Мета**: повністю перенести поточну логіку Telegram-бота з Google Apps Script + Google Sheets у веб‑додаток на **Next.js**, з підтримкою **кількох шкіл**, при цьому:
- основним джерелом даних стає реляційна база (PostgreSQL/MySQL);
- для кожної школи залишається **власна Google Spreadsheet**, куди синхронізуються заявки (аналог поточного листа «Заявки»), щоб менеджери могли й надалі працювати в знайомому інтерфейсі таблиць.

Кожна школа має:
- власні курси;
- власні налаштування (чат/канал для заявок, тексти повідомлень, шаблони питань);
- власні ключі/токени (Telegram Bot Token, API ключ Нової Пошти, тощо);
- ізольовані дані студентів, заявок і відгуків.

Google Sheets замінюється на **реляційну базу даних** (наприклад, PostgreSQL), а логіка тригерів та перевірок переноситься у бекенд Next.js.

---

### 2. Архітектура системи

- **Next.js застосунок** (App Router):
  - Публічні API‑роути для інтеграції з Telegram (webhook) та Новою Поштою.
  - Адмін‑панель (SSR / CSR) для користувача системи.
  - Сторінки для перегляду заявок, фільтрації, експорту, управління курсами та налаштуваннями.
  - Для розділу `Заявки` підтримуються 2 представлення: `Table view` і `Kanban view`.
- **База даних (PostgreSQL / MySQL)**:
  - Зберігає всі сутності, що раніше були в Google Sheets:
    - школи;
    - курси;
    - завдання/вимоги до скрінів;
    - заявки студентів;
    - скріни, оцінки, відгуки;
    - статус обробки, дні на відправку, тощо.
- **Telegram‑боти (мультибот‑архітектура)**:
  - Для **кожної школи** створюється **окремий Telegram‑бот** з власним `BOT_TOKEN`, який працює паралельно з іншими.
  - Усі боти налаштовані на **один спільний webhook‑ендпойнт** бекенда, де школа визначається за конфігурацією бота.
  - Логіка діалогу (питання 1–11) однакова для всіх ботів, але всі дані та налаштування беруться з контексту конкретної школи.
- **Інтеграція з Новою Поштою**:
  - Сервісний модуль у бекенді (wrapper над API Нової Пошти).
  - Пошук міста й відділення за введеним текстом (згідно техзавдання).

---

### 3. Модель даних (скорочено)

**Основні таблиці**:
- `schools` — школи:
  - `id`, `name`, `slug`, `school_key` (публічний ключ/код школи для посилань),
  - `telegram_chat_id` (куди надсилати заявки),
  - `telegram_bot_token`,
  - `nova_poshta_api_key`,
  - `google_sheet_id` / `google_sheet_url` — посилання/ID Google Spreadsheet школи, куди дублюються заявки,
  - інші параметри за потреби.

- `courses` — курси школи:
  - `id`, `school_id`,
  - `title`,
  - `certificate_type` (електронний / фізичний / обидва),
  - `days_to_send` (термін у днях, аналог стовпця E),
  - `review_link` (аналог F2, посилання на незалежний сайт),
  - `requirements_text` (що потрібно надіслати, аналог колонки C).

- `applications` — заявки студентів (аналог листа "Заявки"):
  - `id`, `school_id`, `course_id`,
  - `date_created`,
  - `telegram_user_id`, `telegram_username`,
  - `student_name_ua`, `student_name_en`,
  - `city`, `branch`,
  - `delivery_type` (електронний/фізичний/обидва, Україна/закордон),
  - `status` (новий, в обробці, підтверджено, відхилено, тощо),
  - `score` (1–10),
  - `feedback_text`,
  - `manager_checked` (булеве, аналог чекбокса),
  - `manager_checked_at`.

- `application_screenshots` — скріни:
  - `id`, `application_id`,
  - `file_id` (Telegram `file_id`, без збереження файлів у зовнішньому сховищі),
  - тип задачі за потреби.

- `application_courses` (якщо один студент у рамках однієї заявки обирає кілька курсів) — зв’язуюча таблиця між заявкою та курсами.

За потреби можна додати таблиці для:
- текстових шаблонів повідомлень для кожної школи (див. нижче);
- журналу нотифікацій;
- адміністраторів/менеджерів з правами доступу.

**Шаблони повідомлень бота** (пер‑школа):
- Окрема таблиця, наприклад, `message_templates`:
  - `id`, `school_id`,
  - `code` (унікальний код шаблону, напр. `start_intro`, `after_confirmation`, `nova_poshta_warning`, `ask_name`, `ask_delivery_type`, тощо),
  - `text` (тіло повідомлення з плейсхолдерами типу `{{ім'я}}`, `{{днів}}`),
  - `description` (людський опис, щоб адмінам було зрозуміло).
- Для **кожного етапу діалогу** (питання 1–11, повідомлення після підтвердження, нагадування тощо) є окремий шаблон, який:
  - зберігається **окремо для кожної школи**;
  - може бути змінений адміністратором школи в адмін‑панелі без змін коду.
  - на старті заповнюється дефолтними текстами 1:1 з `docs/work-scope.md`.

**Ролі доступу**:
- У системі використовується **одна роль** — `user`.
- `user` має повний доступ: створення шкіл, редагування шкіл, керування ботами, курсами, шаблонами повідомлень, заявками та синхронізацією.

### 4. Логіка Telegram‑ботів у Next.js

Кожна школа має свій окремий бот, але всі боти працюють через спільний **webhook**:
- Наприклад, `/api/telegram/webhook`, куди Telegram надсилає апдейти для всіх ботів.
- За `BOT_TOKEN` / `bot_username` з апдейта бекенд знаходить відповідний запис школи.
- Для кожного апдейта бот:
  1. Знаходить школу (за ID бота / `BOT_TOKEN`).
  2. Визначає поточний стан діалогу користувача (state‑машина в БД: таблиця `user_sessions` або поле в `applications`).
  3. Відповідає згідно зі сценарієм питань 1–11 з техзавдання.

**Єдиний підхід для retry Telegram**:
- Обробка апдейтів має бути ідемпотентною: кожен `update_id` для конкретного бота обробляється рівно один раз.
- Перед бізнес-логікою бекенд перевіряє, чи `update_id` вже оброблено; якщо так, повертає успіх без повторної обробки.
- Для тимчасових помилок (БД, Google Sheets, зовнішні API) використовуються повторні спроби з backoff.
- Відправка повідомлень користувачу/в чат школи також має ключ ідемпотентності, щоб уникнути дублювань при retry.

**Кроки діалогу (дефолтні тексти 1:1 з `docs/work-scope.md`):**

1. **Питання 1 – старт**  
   - Текст: `Щоб ввести дані для отримання сертифікату натисніть "Старт"`.
   - Кнопка: `Старт`.

2. **Питання 2 – вибір курсу**  
   - Текст 1: `Вітаємо Вас із закінченням навчання`.
   - Текст 2: `Підкажіть, будь ласка, з якого курсу бажаєте отримати сертифікат?`.
   - Кнопки: список курсів поточної школи.

3. **Питання 3 – вимоги + скріни**  
   - Текст: `Що потрібно для отримання Ваших документів:`.
   - Далі додається `requirements_text` курсу: `{текст із таблиці} . Надішліть скрін, будь ласка`.
   - Після кожного скріну: `Надіслати ще` або `Далі`.

4. **Питання 4 – тип сертифікату**  
   - Якщо тільки електронний:
     - `Дуже радіємо, що Ви пройшли цей шлях разом із нами 🥹`
     - `З цього курсу передбачено електронний сертифікат`
   - Якщо фізичний/обидва:
     - `Дуже радіємо, що Ви пройшли цей шлях разом із нами 🥹`
     - `Оберіть формат сертифіката 👇`
     - кнопки: `📦 Фізичний`, `📄 Електронний`, `📦+📄 Обидва`
   - Після вибору: `➕ Обрати ще курс` або `➡️ Перейти далі`.

5. **Питання 5 – ПІБ українською**  
   - Текст: `Введіть Ваше ПІБ українською мовою`.

6. **Питання 6 – ПІБ англійською**  
   - Текст: `Введіть Ваше ім'я та прізвище англійською мовою`.

7. **Питання 7 – доставка (тільки якщо є фізичний сертифікат)**  
   - Текст: `Куди потрібно відправити сертифікат?`.
   - Кнопки: `🇺🇦 По Україні` / `🌍 За кордон`.
   - Для України: інтеграція з API Нової Пошти (місто + відділення).
   - Для закордону:
     - `👉 Для доставки за кордон пропонуємо електронний сертифікат (PDF) 💌`
     - `Він має підпис і печатку та є повноцінним документом. Його можна роздрукувати за потреби.`
     - `Це швидше, зручніше та без ризиків доставки.`
     - `📦 Також, за вашим бажанням, можемо надіслати паперовий сертифікат Новою поштою за умови повної передоплати доставки.`
     - кнопки: `Електронний` / `Фізичний`
     - якщо `Фізичний` за кордон: `Напишіть, будь ласка, такі дані для створення ТТН📦 і наш менеджер зв'яжеться з вами:` + `ПІБ латиницею`, `Країна`, `Місто`, `Адреса / відділення`, `Телефон`, `Email`.

8. **Питання 8 – оцінка**  
   - Текст 1: `Щиро дякуємо, що обрали нашу школу 🥹`.
   - Текст 2: `Наскільки ймовірно, що ви будете рекомендувати наше навчання?`.
   - Кнопки оцінки: `1..10`.

9. **Питання 9 – відгук (опціонально)**  
   - Текст 1: `Наша команда працює 24/7, аби наше навчання було якісним та професійним, і тут нам дуже важливий ваш зворотній зв'язок та ваша думка для того, аби покращувати якість навчання та давати саме ті знання, які допоможуть нашим студентам досягати поставлених цілей)`.
   - Текст 2: `Які у Вас враження від курсу?`.
   - Текст 3: `Чим навчання було корисним для Вас?`.
   - Дія: можна пропустити.

10. **Питання 10 – підтвердження**  
   - Текст: `Уважно перевірте введені дані 👇`.
   - Бот показує сформоване повідомлення з внесеними даними для перевірки.
   - Кнопки: `Видалити` / `Надіслати`.

11. **Питання 11 – завершення**  
   - Після підтвердження бот:
     - створює заявку(и) в БД (по одному запису на кожен курс);
     - надсилає 1-е повідомлення: `Ваші дані для отримання сертифікату з курсу "Назва курсу" / сертифікатів з курсів "Назва курсу - 1", "Назва курсу - 2" ... прийняті в обробку. Після перевірки інформації ви отримаєте відповідь.`
     - надсилає 2-е повідомлення: `Будь ласка, очікуйте повідомлення протягом доби. 😊`
     - якщо оцінка = 10, надсилає 3-є повідомлення: `Будемо вдячні, якщо ви залишите свій відгук також на незалежному сайті 👉 Залишити відгук (зашити посилання з клітинка F2) Ваш досвід може стати корисним і надихнути інших 🫶🏼`.

### 5. Надсилання в Telegram‑чат школи

Замість прямого запису в Google Sheets:
- Після створення заявки в БД бекенд формує **одне повідомлення** з усіма даними та списком скрінів.
- Надсилає його в Telegram‑чат школи за `telegram_chat_id` (аналог A2 з листа «Налаштування», але тепер на рівні таблиці `schools`).
- У повідомлення можна включити:
  - ПІБ, курси, тип сертифікату;
  - місто/відділення або закордонну адресу;
  - оцінку та відгук;
  - посилання для відкриття заявки в адмін‑панелі.

---

### 5.1. Синхронізація заявок у Google Spreadsheet школи

Щоб зберегти звичний для клієнта формат роботи з таблицями:
- Для **кожної школи** вказується своя Google Spreadsheet (`google_sheet_id` / `google_sheet_url`) з листом «Заявки» (або іншим, узгодженим форматом).
- Після створення/оновлення заявки в БД:
  - бекенд створює/оновлює відповідний рядок у таблиці Google Sheets цієї школи;
  - у рядку містяться основні поля заявки (ПІБ, курси, дата, статус, тип сертифікату, місто/відділення, оцінка, відгук, посилання на адмін‑панель тощо).
- Синхронізація реалізується через Google Sheets API з сервісним акаунтом:
  - один сервісний акаунт із доступом до всіх шкільних таблиць, або
  - окремі налаштування доступу на рівні кожної школи (за потреби).

Політика синхронізації:
- Потрібна **1:1 синхронізація** між БД і Google Sheets для кожної школи.
- Кожен запис заявки має стабільний `external_row_id` (або еквівалент), щоб однозначно мапити запис БД на рядок у таблиці.
- Будь-яка зміна в БД має бути відображена в Google Sheets, і зміна у Google Sheets (якщо дозволено) має бути відображена в БД.
- Для уникнення розсинхронізації використовується черга синхронізації з повторними спробами, статусом синку та журналом помилок.

### 6. Логіка підтвердження менеджером (аналог чекбокса)

Замість чекбокса в Google Sheets:
- В адмін‑панелі Next.js (для менеджерів школи) є список заявок.
- Менеджер може:
  - відмітити заявку як «перевірено/підтверджено»;
  - змінити статус (наприклад: «Відправлено», «Виготовляється», тощо);
  - переглянути всі дані та скріни.

**Після встановлення статусу «підтверджено»:**
- Бекенд надсилає користувачу в Telegram:
  - повідомлення з шаблону `after_confirmation`; дефолтний текст 1:1:
    - `Доброго дня, {{Ім'я}} 🫶`
    - `Дані для отримання сертифіката перевірено та прийнято✅`
    - `Очікуйте оформлення та відправку Ваших документів протягом {{днів}} днів✉️`
  - друге повідомлення з шаблону `nova_poshta_warning`; дефолтний текст 1:1:
    - `☝️Відповідальність за стан документів під час доставки несе Нова пошта.`
    - `У разі пошкодження, будь ласка, оформіть акт у відділенні та надішліть нам його фото — ми здійснимо повторне відправлення 💌`
    - `*Повторне відправлення можливе й без акту, однак акт необхідний для компенсації за очікування та незручності📦`

Всі інші службові тексти бота (вступні повідомлення, запити ПІБ, інструкції щодо скринів, подяка за відгук тощо) також реалізуються через таблицю `message_templates`:
- адміністратор кожної школи може в адмін‑панелі змінювати тексти для **кожного етапу діалогу**;
- бекенд підставляє актуальний текст для цієї школи при формуванні повідомлень у боті.

### 7. Мультишкільність і розмежування даних

- Кожен користувач Telegram всередині бота **прив’язується до конкретної школи** через окремий бот цієї школи (окремий `BOT_TOKEN` / `bot_username` для кожної школи).
- Усі запити, збереження й вивід даних завжди фільтруються по `school_id`.
- Адміністратори/менеджери школи:
  - мають кабінет з авторизацією (наприклад, через email+пароль або OAuth);
  - бачать тільки дані своєї школи.
---

### 8. Технологічний стек Next.js

- **Runtime / Package manager**: `Bun`.
- **Next.js 14+ (App Router)**:
  - API‑роути для Telegram/Нова Пошта/внутрішній API.
  - SSR/ISR для публічних сторінок (якщо будуть).
- **База даних**: PostgreSQL + Prisma ORM.
- **Авторизація в адмін‑панелі**: `better-auth` (email + password).
- **UI/UX**:
  - `shadcn/ui` як базова бібліотека компонентів;
  - таблиці для списків заявок/шкіл/курсів на базі DiceUI Data Table (Radix): https://www.diceui.com/docs/components/radix/data-table;
  - додатково для `Заявки`: Kanban-представлення на базі DiceUI Kanban (Radix): https://www.diceui.com/docs/components/radix/kanban;
  - Tailwind CSS теми: `light` та `dark`.
- **Файли скрінів**:
  - зберігати тільки в Telegram через `file_id` (без S3 і без завантаження на свій сервер).

---

### 9. Підсумок

Запропонована архітектура на **Next.js**:
- переводить бізнес‑логіку з Google Apps Script у власний бекенд і БД;
- підтримує **кілька шкіл** з ізольованими курсами, заявками, ключами й налаштуваннями;
- дозволяє **кожній школі прив’язати власну Google Spreadsheet**, куди автоматично синхронізуються заявки для подальшої роботи менеджерів;
